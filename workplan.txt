============================================================
STEP 1 — CI/CD & Monorepo tooling direction
============================================================
- Build a generic GitHub Actions setup using reusable actions/workflows where possible
  (shared across backend/frontend rather than duplicated per-package pipelines).
- Monorepo tooling: NX was evaluated (see 1a below) and dropped as overkill for 2
  unrelated projects — kept the npm workspace (single lockfile) without NX on top.
- Push built images to Amazon ECR.
- Use OIDC (GitHub Actions OIDC provider -> AWS IAM role) for authentication to AWS,
  instead of long-lived AWS access keys stored as secrets.

Sub-step order decided (2026-07-05):
  1a. NX monorepo setup for backend/frontend (done first — changes how tasks are
      invoked, so the CI refactor in 1b is written against NX from the start).
      [Revised 2026-07-06: NX removed after trying it, see below — 1a is now just
      the npm workspace, no NX.]
  1b. GitHub Actions reusable-workflow refactor: two independent workflows
      (backend, frontend), sharing logic via reusable workflows/composite actions
      where it makes sense, instead of NX task orchestration.
  1c. Bootstrap Terraform (small, separate stack, e.g. infra/bootstrap/) — creates
      just the ECR repositories + GitHub OIDC provider + IAM role needed for CI.
      This is intentionally separate from Step 4's full VPC/EKS/RDS Terraform,
      since the CI/CD tooling has to exist before the "real" infra it would
      normally help deploy.
  1d. Wire the GitHub Actions workflow to assume the IAM role via OIDC and push
      images to ECR.

STEP 2 — Umbrella Helm chart
============================================================
- Convert the current single Helm chart (helm/calendar/) into a generic umbrella
  chart: a top-level parent chart with "frontend" and "backend" as subcharts
  underneath it.
- Each subchart owns its own templates/values (deployment, service, etc. currently
  under helm/calendar/templates/backend-*.yaml and frontend-*.yaml would move into
  helm/calendar/charts/backend/ and helm/calendar/charts/frontend/ respectively).
- Parent chart's values.yaml holds shared/global config and per-subchart overrides
  (e.g. global.image.registry, global.env), subcharts expose their own defaults.
- Shared resources that don't belong to either service (ingress, migration-job,
  any cross-cutting config) stay at the umbrella/parent level.

STEP 3 — Argo CD / GitOps deployment
============================================================
- Deploy the umbrella chart (from Step 2) via Argo CD rather than manual
  helm install/upgrade.
- Use an ApplicationSet to generate Argo CD Applications (e.g. per
  environment/cluster, or per subchart) instead of hand-written Application
  manifests per target.
- Consider an "app of apps" pattern: a root Argo CD Application that points at
  a repo/dir of child Application manifests, so the whole environment is
  bootstrapped from one entry point.
- Full GitOps model: Git is the source of truth for desired state; CI (Step 1)
  builds/pushes images and updates the manifest/values repo (e.g. bumping an
  image tag), Argo CD detects the change and syncs the cluster — no direct
  `helm upgrade`/`kubectl apply` from CI.

Decisions made (2026-07-06):
  - Argo CD install: official kubectl quickstart manifest (not Helm) - simplest
    first bootstrap, no self-management benefit yet on one local cluster.
  - Repo structure: keep app code + Helm chart in this one repo. Argo's
    Application only diffs the helm/calendar path specifically, so app-code
    commits elsewhere are harmless no-ops for it - matches the bootcamp
    material's own "small/solo project" guidance, not the "mixing code and
    config" anti-pattern it warns larger orgs about.
  - App-of-Apps is genuinely justified here (not just for show): a root
    Application will manage two real children - the calendar app now, and
    kube-prometheus-stack once Step 5 adds it. That's the actual two-app case
    the pattern is for.
  - ApplicationSet for the calendar app: List generator with ONE element today
    (env: kind, values file: local-dev/values-kind.yaml), structured so adding
    a second environment later is a one-line list addition - honest that this
    is "shape it right for later," not "we need this at 1 app" (ApplicationSets
    are for >20 similar apps per the bootcamp material; we have 1).
  - Sync policy: full automation (automated + selfHeal + prune) for this local
    learning cluster, unlike a real prod setup that would likely gate on
    manual approval.
  - Known, deliberate gap NOT solved in this step: Helm values still pin
    `tag: latest` (a moving pointer), not a per-commit SHA that CI bumps in
    Git - true "Git = exact deployed version" GitOps needs CI to write the
    SHA into a tracked values file on every push. Real, separate follow-up
    work, not silently bundled into Step 3.

STEP 4 — Terraform: VPC, EKS, cluster add-ons, RDS
============================================================
- Provision networking and compute with Terraform, built from modules (prefer
  well-maintained community modules, e.g. terraform-aws-modules/vpc and
  terraform-aws-modules/eks, over hand-rolled resources) rather than one flat
  root module.
- VPC: public/private subnets across multiple AZs, NAT gateway(s), private
  subnets for worker nodes and RDS.
- EKS cluster with:
  - Cluster Autoscaler (or Karpenter as a modern alternative — worth deciding
    between the two) for node scaling.
  - AWS Load Balancer Controller (ALB controller) to provision a cloud ALB/NLB
    from a Kubernetes Ingress/Service of type LoadBalancer.
  - VPC CNI addon (pod networking) and EBS/EFS CSI driver addon (persistent
    storage) — install as managed EKS add-ons where possible.
  - An Ingress controller (via the ALB controller above) so Kubernetes Ingress
    resources provision a real cloud load balancer.
- RDS for the Postgres database instead of the in-cluster/Docker Compose
  Postgres used today — best practices: private subnet only (no public
  access), Multi-AZ for prod, encrypted storage, credentials via AWS Secrets
  Manager (or SSM Parameter Store) rather than plain Terraform variables,
  automated backups/snapshots, parameter group tuned for the workload.
- Terraform best practices to apply: remote state (S3 backend + DynamoDB lock
  table), separate state/workspaces per environment, module versioning, least-
  privilege IAM (ties into Step 1's GitHub OIDC role and the ALB
  controller/CSI/autoscaler each needing their own scoped IAM roles via IRSA).

Decisions made (2026-07-08):
  - Cost is the dominant constraint here, unlike every step before it: an
    always-on EKS+RDS+NAT setup runs ~$150-300+/month, vs Step 1c's
    essentially-free IAM/ECR bootstrap. User wants to apply-and-destroy every
    working session to minimize spend, not leave this running.
  - That workflow requires state to survive the destroy/apply cycle, so this
    splits into infra/state-backend/ (tiny, permanent, applied once, never
    destroyed - S3 bucket + DynamoDB lock table) and infra/main/ (the actual
    expensive VPC/EKS/Karpenter/RDS stack, meant to be destroyed regularly).
  - Karpenter over Cluster Autoscaler - bootcamp material's own framing as
    the modern AWS-native approach.
  - Cost-optimized (not textbook-HA) sizing, explicitly agreed given the
    apply/destroy workflow: single NAT gateway (not per-AZ), single-AZ RDS
    (not Multi-AZ), small/spot Karpenter instances.
  - RDS data is ephemeral by design (won't survive a destroy) - acceptable,
    matches how kind's local Postgres is already treated.

4e IN PROGRESS (blocked on AWS account verification, 2026-07-08) - AWS Load
   Balancer Controller + ingress-nginx (NLB, not ALB):
  - Switched the ingress design from a plain ALB Ingress to NLB (Layer 4) +
    ingress-nginx (in-cluster Layer 7) - decided after discussing the real
    trade-off: NLB preserves real client source IPs (ALB doesn't, by
    default), ingress-nginx has a much richer feature set than raw ALB
    Ingress annotations, and this keeps helm/calendar/templates/ingress.yaml
    portable (plain ingressClassName: nginx, no AWS-specific annotations
    baked into the app chart itself) vs. locking it to EKS. The AWS Load
    Balancer Controller is still required even for this NLB path - it's what
    actually understands the aws-load-balancer-* Service annotations for
    IP-target-mode NLBs (the legacy in-tree cloud provider can only do
    "instance" target mode, which is fragile against Karpenter dynamically
    creating/destroying nodes).
  - helm/calendar/templates/ingress.yaml reworked: host and annotations both
    now optional (only rendered if set) since this project has no owned
    domain - reached via the NLB's own AWS-generated hostname, not a real
    domain, so no TLS either. helm/calendar/values.yaml ingress.className
    changed from alb to nginx to match.
  - Copied the "official" AWS Load Balancer Controller IAM policy from the
    bootcamp reference material into infra/main/ingress-nginx.tf as a
    starting point - it needed two real, live-debugged corrections, both
    now fixed (same "reference material is version-drifted" pattern hit
    repeatedly in 4c/4d):
    - Issue 1: AccessDenied on elasticloadbalancing:AddTags immediately
      after the controller created a target group. The copied statement had
      a strict conditional (only allows AddTags once a resource is ALREADY
      tagged as cluster-owned) that doesn't hold at the moment of resource
      creation with this controller version (3.4.1). Fixed by simplifying
      to an unconditional Allow on the specific target-group/load-balancer
      ARNs - a deliberate least-privilege trade-off, not a full revert to
      wildcard-everything.
    - Issue 2: separately, ec2:DeleteSecurityGroup was missing from the
      policy ENTIRELY (not just over-conditioned) - only surfaced once the
      controller tried to clean up the security group it had created, during
      an uninstall/replace. This left a real orphaned Kubernetes Service
      object stuck with a `service.k8s.aws/resources` finalizer that could
      only be cleared once the permission existed AND the controller pod was
      manually restarted (`kubectl delete pod ...`) to force an immediate
      reconcile instead of waiting out its retry backoff - the fix alone,
      without the restart, would not have unstuck the already-orphaned
      Service on its own.
  - After both real bugs were fixed, hit a genuinely different, unrelated
    error: `OperationNotPermitted: This AWS account currently does not
    support creating load balancers. ... contact AWS Support` on
    CreateLoadBalancer - confirmed via `aws elbv2 describe-load-balancers`
    returning zero existing LBs (ruled out a quota-from-existing-resources
    explanation). Root cause identified: this is a brand-new AWS account,
    confirmed via the Cost Management console's first-time "check back in
    24 hours" message - AWS gates ELB/NLB/ALB creation behind an automated
    new-account verification window (commonly 24-48h) that isn't visible or
    fixable via the API/Terraform/IAM at all; a payment method was already
    on file, and no Budget Action/SCP was involved (ruled both out - the
    error text itself is ELB-service-specific, not an IAM/SCP-style
    AccessDenied). Nothing left to fix in code - infra/main/ingress-nginx.tf
    is believed correct and ready to re-apply once the account restriction
    clears.
  - Session ended with `terraform destroy` of the whole infra/main/ stack
    (cost control, as agreed for every session) rather than leaving it
    running idle while blocked on an external account-verification wait.
